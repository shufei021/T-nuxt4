# 架构说明

## 目标

本模板用于构建标准化的 Nuxt 4 前端项目，适配前后端分离团队协作模式。

核心目标：

- 前端聚焦页面、交互、状态和工程规范
- 后端聚焦业务接口、权限、数据和服务稳定性
- 前后端通过清晰的 API 契约协作

## 分层建议

### 页面层 `app/pages`

负责：

- 页面结构
- 页面交互
- 页面级数据装配

不建议：

- 在页面里直接散落大量请求代码
- 在页面里硬编码接口地址
- 在页面里混入复杂的数据转换逻辑

### 组件层 `app/components`

负责：

- 可复用展示组件
- 通用 UI 结构
- 局部交互封装

### 组合层 `app/composables`

负责：

- 请求封装
- 可复用业务逻辑
- 页面逻辑复用

例如：

- `useApi` 统一请求入口
- `useHealthCheck` 统一健康检查逻辑

### 状态层 `app/stores`

负责：

- 全局状态
- 多页面共享状态
- 应用级缓存状态
- 登录态与权限状态

### 类型层 `app/types`

负责：

- 后端响应类型
- 业务实体类型
- 请求参数类型

### 中间件层 `app/middleware`

负责：

- 登录态守卫
- 游客页拦截
- 角色权限校验

模板默认提供：

- `auth.ts`
- `guest.ts`
- `permission.ts`

## 请求流转建议

推荐调用路径：

`Page -> Composable -> useApi -> Backend API`

这样做的好处：

- 错误处理集中
- Token 管理集中
- 请求头策略集中
- 更容易接入埋点、重试、刷新 token

## 环境配置建议

建议至少维护以下环境：

- 本地开发
- 测试环境
- 预发布环境
- 生产环境

统一通过环境变量提供：

- `NUXT_PUBLIC_API_BASE`
- `NUXT_PUBLIC_APP_TITLE`

## 不建议的做法

- 在组件中直接拼接完整后端域名
- 每个页面都重复写一套请求逻辑
- 后端返回结构不统一
- 接口错误码没有统一约定

## 推荐演进方向

随着项目规模扩大，可以继续加入：

- 通用表格查询方案
- 上传下载封装
- 国际化
- 监控与埋点
- 自动化测试
- OpenAPI SDK 生成
